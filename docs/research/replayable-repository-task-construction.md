# Existing Research on Automatically Constructing Replayable Repository-Specific Software Engineering Tasks from Repository History

## Where the literature is closest to this problem

The closest existing line of work is not classic static coding benchmarks, but repository-level benchmarks and data-construction pipelines that mine real issues, pull requests, commits, security patches, or CI failures from entity["company","GitHub","developer platform company"] repositories, then reconstruct a historical repository state and validate candidate tasks by replaying builds or tests. *SWE-bench* established the canonical issue-to-patch formulation; later work automated historical environment setup to scale task extraction beyond a few manually curated projects; still newer work mined merged PRs, vulnerability-fixing commits, regression-inducing histories, and CI failures instead of starting only from issue text. citeturn6view0turn31view3turn31view5turn8view6turn36view0turn37view0

There is also an adjacent but less directly aligned strand that auto-generates feature tasks or mirrors existing fixes into new tasks. That work is useful because it shows how to synthesize prompts, tests, and executable environments at scale, but it is weaker as evidence of what historically happened in a given repository, because the task itself is partly synthesized rather than directly replayed from one repository’s own evolution history. citeturn29view6turn29view7turn30view0

## Method families in current work

The first major family is **issue-to-patch** and its close relative **issue-to-test**. In *SWE-bench*, the model sees an issue description and the pre-fix repository snapshot, while correctness is judged by hidden developer-written tests associated with the historical fix. *SWE-bench-Live* keeps the same basic task but automates issue–PR discovery and instance refresh to reduce staleness and contamination. *TDD-Bench-Verified* reuses the same historical issue-resolution setting for a different task: generate tests from the issue and old code, then check whether those tests fail on the old code and pass on the hidden post-fix code. *Otter* and *blast* build on this setting by using historical issue or issue-patch pairs to generate issue-reproducing tests. citeturn6view0turn30view1turn35view0turn35view1turn33view0

The second major family is **PR-to-task** and **commit-to-task**. The 2025 work on automated benchmark generation with *SETUPAGENT* defines a “valid PR” as one that is merged, resolves an issue, and modifies a test file; it then splits the PR into a reference code patch and a test patch and keeps only instances with real fail-to-pass behavior. *FEA-Bench* mines feature-oriented PRs, combines rule-based filtering with LLM-based intent classification, and explicitly stores the base commit, environment settings, gold patch, and test patch. *SWE-Next* derives a pair \((c_{base}, c_{merged})\) from each merged PR and uses execution-grounded filtering to keep only reproducible tasks. The time-consistent PR methodology snapshots a repository at time \(T_0\) and converts later merged PRs into task descriptions while preventing post-\(T_0\) leakage. *SWE-CI* goes even further from issue-resolution and treats a base/oracle commit span as the task, evaluating whether an agent can maintain and evolve code over a substantial stretch of history. Commit-centric variants include *RegMiner*, which mines regressions by starting from bug-fixing commits with added tests and tracing back to regression-inducing commits; *SecRepoBench*, which mines vulnerability-fixing commits and masks the patched region to create a repository-level security task; and *ComBench*, which mines real CI compilation failures and their repair patches from repository CI history. citeturn31view0turn31view1turn29view1turn29view2turn29view4turn7view7turn7view8turn6view4turn7view11turn9view0turn9view1turn9view2turn9view3turn10view0turn10view1turn36view0turn36view4turn37view0

A third family supplies the **mining machinery** that makes the previous families work. For issue/commit traceability, *ReLink* recovers missing links from temporal proximity, textual similarity, and committer–owner correspondences; *Hybrid-Linker* combines textual and non-textual features; and *EasyLink* uses embedding-based retrieval plus LLM reranking, showing that stronger modern IR baselines materially change results. For bug-fix and bug-introducing commit identification, SZZ-style pipelines remain central: *SZZ Unleashed* formalizes the workflow of extracting bug reports, linking them to bug-fixing commits, and tracing candidate bug-introducing commits; *PR-SZZ* extends that workflow with PR graphs and inner-commit reasoning; and developer-informed oracle work improves evaluation by using commits where developers explicitly named the inducing change. A 2020 traceability review also found that most change-impact traceability studies still rely on IR-style recovery and that dataset scarcity remains a structural limitation. citeturn15view0turn15view1turn15view3turn16view0turn14view6turn14view7turn14view9turn6view7turn17view1turn17view3turn17view4turn13view0turn13view4turn21view0

## The common pipeline in prior literature

Across the history-based papers, the pipeline is surprisingly stable. First, repositories are filtered for executability, licensing, and enough activity or test infrastructure to make replay feasible. Second, candidate changes are mined from issue-linked PRs, merged PRs, bug-fixing commits that add tests, vulnerability-fixing commits, or CI failures. Third, the system chooses a replay point: the code right before a fix, the target branch state right before PR merge, a failing CI revision, or the “base” endpoint of a longer base/oracle pair. Fourth, it reconstructs a historical environment using repository docs, dependency manifests, lockfiles, or CI configuration; this step is explicit in *SETUPAGENT*, *SWE-CI*, and *ComBench*, and it is repeatedly described as the core bottleneck for scale. Fifth, it turns the historical change into evaluation artifacts by separating non-test edits from test edits, or by masking a patched region while preserving the rest of the repository. Sixth, it runs tests or builds under multiple states and keeps only candidates with unambiguous transitions such as fail-to-pass, build-fail-to-build-pass, or vulnerable-to-secure. Finally, some newer papers synthesize a natural-language task statement from issues, PRs, tests, and patches, but only after execution-grounded validation has already succeeded. citeturn31view3turn31view5turn29view2turn29view4turn8view6turn9view0turn9view1turn9view2turn37view0turn36view0turn36view4turn30view0turn6view4

What changes from paper to paper is therefore less the overall pipeline than the **candidate source** and the **oracle type**. Issue-centric pipelines inherit a natural-language problem statement “for free” but depend heavily on link recovery and test quality. Commit-centric pipelines are often cleaner for replay and patch extraction, but they may need stronger prompt synthesis because the historical commit message or PR description is not always a good standalone task statement. CI-centric pipelines yield authentic operational failures and reproducible diagnostics, but they skew toward build, compilation, and dependency-related work rather than arbitrary bug fixes or features. citeturn6view0turn29view1turn37view0turn36view0

## How inputs tests and gold patches are constructed

Existing work builds the **problem statement** in three main ways. The highest-fidelity approach is to reuse the original issue text, as in *SWE-bench*-style benchmarks. PR-oriented feature benchmarks use PR descriptions and structured extraction of new components or requirements; *FEA-Bench* is a clear example. A more synthetic variant constructs a fresh task statement from multiple historical artifacts: the time-consistent benchmark transforms historical PRs into natural-language tasks with an LLM-assisted prompt-generation pipeline, and *SWE-Mirror* synthesizes a self-contained issue-like statement from the original issue, the generated test patch, and the fix patch. **Base commit** selection is likewise standardized: papers use the repository state immediately before the fix, the pre-merge target-branch state, an earlier commit in a regression trace, the vulnerable state before a security fix, or a failing build snapshot from CI history. citeturn6view0turn29view1turn29view4turn7view11turn30view0turn10view0turn36view0turn37view0

The task input almost always includes **executable context**, not just text. *FEA-Bench* stores environment setup information and base commit metadata. The *SETUPAGENT* paper is especially explicit that historically accurate replay requires extracting installation commands and test commands from repository documentation and, if needed, external documents. Hidden evaluation usually comes from developer-written test patches or other opaque executables: SWA-style datasets split each PR into code patch and test patch; *FEA-Bench* stores test patches plus fail-to-pass and pass-to-pass lists; *TDD-Bench-Verified* hides the new code rather than the tests because the task is to generate tests; and *SecRepoBench* combines developer-written unit tests with external security test cases from OSS-Fuzz. citeturn29view2turn29view4turn29view9turn31view3turn31view5turn35view0turn36view3

The **gold patch** is used in at least four distinct ways in the literature. First, it is the reference implementation or oracle endpoint. Second, it defines the expected hidden-test behavior: in SWA- and FEA-style pipelines, the test patch is applied around the base snapshot and the gold code patch must flip at least one relevant test from failing to passing. Third, several papers use the gold patch to estimate difficulty, with measures such as edited lines, edited files, or the number of flipped tests. Fourth, gold patches support validation beyond pure execution: developer-informed SZZ evaluation mines commit messages in which developers explicitly named inducing commits; *SecRepoBench* uses vulnerability-fixing commits to locate the masked region and ground the secure behavior; and *ComBench* and *SWE-CI* treat the developer’s repaired endpoint as the oracle state. citeturn8view0turn8view4turn31view1turn29view2turn29view9turn13view0turn13view4turn36view0turn36view4turn37view0turn9view0turn9view3

## What makes a historical change suitable

A historical change becomes a usable benchmark task only when the literature can make it **replayable, testable, and discriminative**. The recurring accept/reject rules are straightforward: the environment must build; the tests or build command must run on both the old and new states; at least one meaningful transition must occur, such as fail-to-pass or build-fail-to-build-pass; flaky instances should be removed; and the change should be neither trivial nor too entangled to localize. Those criteria appear, in different forms, in *SETUPAGENT*, *SWE-Next*, *TDD-Bench-Verified*, *SWE-Mirror*, *SWE-CI*, *RegMiner*, and *ComBench*. citeturn31view1turn31view3turn31view5turn7view7turn35view0turn30view0turn9view1turn9view2turn10view0turn10view2turn37view0

The hardest failure modes are usually **not** patch extraction itself. They are missing or noisy links, noisy bug labels, environment drift, CI noise, and tangled multi-commit changes. Older work reported that 54% of fixed bugs in bug databases were not linked to change logs, and a later large-scale platform study reported only 42.2% of issues manually linked to commits. SZZ-style fixing-commit identification is also noisy: a widely cited empirical study found that only about half of the commits SZZ labels as bug-fixing are truly bug-fixing, and that downstream release-level labels can be badly distorted. Environment setup is another major bottleneck: both the automated benchmark-generation line and CI-mined benchmarks emphasize that historical dependency setup, command discovery, and log parsing are the rate-limiting steps for scaling beyond manually curated repositories. This aligns with a 2022 systematic review of mining software repositories, which highlighted dataset selection and reproducibility as recurring problems in MSR workflows. citeturn15view0turn16view0turn12view0turn31view3turn31view5turn37view0turn38search2

A newer problem is **evaluation leakage and test overfitting**. Time-consistent and live-benchmark papers treat pre/post-snapshot separation as essential because otherwise later code or text can leak the answer into retrieval or prompting. *SecRepoBench* explicitly applies semantic-preserving mutation to reduce literal memorization of historical patched code. Test-generation and evaluation papers also now warn that fail-to-pass behavior is not a complete specification and that repository-level benchmarks can be overfit to visible or weak tests, even when hidden tests exist. citeturn6view4turn30view1turn36view2turn33view0turn34search9

## Most transferable ideas and representative references

For the specific scenario “given one repository, generate repository-specific benchmark tasks from that repository’s own history,” the most transferable literature is the **execution-grounded PR/commit mining** line, not the fully synthetic task-generation line. The strongest matches are *SETUPAGENT* and related automated issue/PR mining, *SWE-Next*’s merged-PR \((c_{base}, c_{merged})\) formulation, the time-consistent PR protocol, *FEA-Bench*’s explicit split between code patch and test patch, and domain-specific commit/CI miners such as *RegMiner*, *SecRepoBench*, and *ComBench*. These approaches preserve repository-specific behavior, keep a replayable old/new state, and retain the human developer patch as the gold oracle. By contrast, synthetic feature generators and mirrored-task frameworks are useful mainly as adjacent evidence about prompt or test synthesis, not as the most faithful basis for repository-historical evaluation. citeturn31view3turn31view5turn8view6turn6view4turn29view2turn29view4turn10view0turn36view0turn36view4turn37view0turn29view6turn30view0

The representative **papers** worth reading first are: *SWE-bench* for the basic issue-to-patch setup; *Automated Benchmark Generation for Repository-Level Coding Tasks* for historical environment automation; *SWE-bench-Live* and *A Time-Consistent Benchmark for Repository-Level Software Engineering Evaluation* for freshness and leakage control; *SWE-Next*, *FEA-Bench*, and *SWE-CI* for PR- and commit-derived task construction; *RegMiner*, *SecRepoBench*, and *ComBench* for regression, security, and CI-failure mining; and *TDD-Bench-Verified*, *Otter*, and *blast* for generating tests from issues or issue-patch pairs. citeturn6view0turn31view3turn30view1turn6view4turn8view6turn28view0turn9view0turn10view0turn36view0turn37view0turn35view0turn35view1turn33view0

The representative **tools and mining methods** are: *ReLink*, *Hybrid-Linker*, and *EasyLink* for issue–commit traceability; *SZZ Unleashed*, developer-informed SZZ evaluation, and *PR-SZZ* for bug-fixing and bug-introducing commit identification; and *Chianti*, *Ekstazi*, *iJaCoCo*, and predictive test selection for impact analysis, test selection, and efficient validation.
