# Repository-Level Code Understanding, Context Selection, and Cross-File Editing

## Scope and framing

The literature most relevant to your project does **not** center on “one fixed benchmark for all coding models.” Instead, it breaks into four partially overlapping lines: repository-level completion, repository-level retrieval and navigation, executable repository modification, and benchmark-generation pipelines that mine tasks from real repository history. The closest prior work to your framing is the small but growing line on **automated benchmark construction** and **live executable evaluation**, including SetUpAgent’s automated generation of SWEE-Bench and SWA-Bench, SWE-bench-Live’s live-updatable issue benchmark, and TimeMachine-bench’s automatically generated migration tasks from dependency breakage. These works matter because they move from hand-curated static suites toward scalable task generation, but they still produce **shared public benchmarks**, not a method for generating a **repository-specific evaluation environment for an arbitrary target repo** with its own conventions, modules, and permissions. citeturn13view8turn35view1turn13view7

A second important framing point is that your project evaluates an **agent configuration** rather than a bare model. Existing papers usually evaluate one of three things instead: a model’s ability to complete code given cross-file context, a retriever’s ability to find relevant repository context, or an end-to-end issue solver’s ability to pass tests. ContextBench is especially relevant because it explicitly argues that end-task success hides what the agent actually did during context retrieval, then introduces process-level metrics over retrieval trajectories. That is unusually close to your concern with whether a particular workflow is trustworthy inside a specific repository. citeturn16view0turn16view3

## Benchmark landscape by capability dimension

**Completion-centered repository understanding.** RepoBench, RepoEval, CrossCodeEval, and ExecRepoBench are the clearest representatives of repository-level **completion** benchmarks. RepoBench evaluates a retrieval task, a next-line completion task, and a retrieval-plus-completion pipeline over Python and Java repositories. RepoEval, introduced with RepoCoder, covers line, API-invocation, and function-body completion. CrossCodeEval creates multilingual cross-file completion tasks that are designed by static analysis to *require* cross-file context. ExecRepoBench pushes in a more executable direction, but it is still fundamentally a completion benchmark, not a realistic repository-edit benchmark. These benchmarks are useful if the question is “can the system exploit repository context to complete local code correctly?” They are much less informative for “can the system plan, localize, edit, verify, and preserve consistency across the whole repository?” citeturn17view5turn18view0turn17view3turn17view6

**Retrieval, navigation, and codebase QA.** RepoQA, ContextBench, CoReQA, CodeRepoQA, SWE-QA-Pro, and wider code-retrieval suites such as CodeSearchNet, GenCodeSearchNet, and COIR probe repository understanding more directly. RepoQA reframes long-context understanding as a realistic code-search-style task: identify the right function from its natural-language description inside a long repository context. ContextBench contributes human-annotated gold contexts and process metrics for retrieval recall, precision, and efficiency over issue-resolution trajectories. CoReQA and CodeRepoQA move into repository-level question answering, where the system must retrieve and synthesize information from the codebase and surrounding discussions. SWE-QA-Pro is notable because it explicitly tries to avoid popularity and memorization bias by using long-tail repositories and filtering out questions solvable by direct answering, showing that agentic exploration matters. CodeSearchNet, GenCodeSearchNet, and COIR are broader code-search benchmarks rather than repository-action benchmarks, but they remain highly relevant for evaluating issue-to-file retrieval and semantic code search components. citeturn24view0turn16view0turn24view1turn24view2turn25view0turn33view0turn33view4turn33view2

**Executable repository modification.** SWE-bench and its successors are the canonical benchmark family for real repository edits derived from real issues and tests. SWE-bench gives the agent a repository and an issue, and judges the output patch by execution against repository tests. SWE-bench-Live extends this idea with fresh post-2024 tasks across many more repositories and automated environment setup, mainly to reduce staleness, overfitting, and contamination. SWE-PolyBench broadens issue-resolution beyond Python to multiple languages and includes bug fixes, feature additions, and refactoring. FEA-Bench focuses specifically on repository-level feature implementation mined from pull requests, while REPOFIXEVAL frames repository-level program repair as issue discovery, fault localization, and patch generation over multi-file repositories. DependEval is different again: it is a hierarchical benchmark of dependency recognition, repository construction, and multi-file editing, intended to expose cross-file reasoning difficulties that end-to-end issue benchmarks blur together. citeturn12view3turn35view1turn13view5turn14view2turn19view5turn15view4

**Migration, modernization, and refactoring.** For your interest in dependency-aware edits, interface migration, API-usage consistency, and refactor consistency, the most relevant benchmarks are newer maintenance-oriented suites. SWE-Refactor mines 1,099 developer-written, behavior-preserving refactorings from 18 Java projects and explicitly includes repository structure, callers/callees, build configuration, and test coverage metadata. MIGRATION-BENCH evaluates repository-level Java migration from Java 8 to newer LTS versions. TimeMachine-bench generates real-world Python migration tasks by reproducing repository failures that arise after dependency updates, and RepoMod-Bench evaluates repository modernization through implementation-agnostic interface testing across 21 repositories in 8 languages. These are among the best current proxies for cross-file consistency under real dependency and interface constraints. citeturn26view3turn10search1turn13view7turn13view6

Taken together, the benchmark landscape answers your specific sub-questions fairly clearly. **Only-completion benchmarks** include RepoBench, RepoEval, CrossCodeEval, and ExecRepoBench. **Benchmarks that test real executable modification** include SWE-bench, SWE-bench-Live, SWE-PolyBench, FEA-Bench, REPOFIXEVAL, SWE-Refactor, TimeMachine-bench, MIGRATION-BENCH, and RepoMod-Bench. **Benchmarks that best reflect repository conventions and engineering constraints** are those grounded in real issues, PRs, tests, build settings, or dependency updates, especially SWE-bench-family tasks, FEA-Bench, SWE-Refactor, TimeMachine-bench, and RepoMod-Bench. **Benchmarks that support differential evaluation across repositories** do so only partially: many include multiple repos and allow per-repo analysis, but most are still designed as fixed shared public suites rather than as a mechanism for generating a distinct benchmark for each target repository. citeturn17view5turn18view0turn17view3turn17view6turn12view3turn35view1turn13view5turn14view2turn19view5turn26view3turn13view7turn13view6turn13view8

## Methods for context selection and file localization

The dominant line of work on **context selection** is still retrieval-augmented generation, but it has evolved from simple embedding lookup into richer multi-stage selection. RepoCoder uses similarity-based iterative retrieval to feed repository context back into completion. Repoformer adds a policy that decides **whether retrieval is needed at all**, showing that selective retrieval can preserve or improve accuracy while reducing inference cost. CodeRAG goes further by improving retrieval query construction, combining multiple retrieval paths, and reranking retrieved context to align it with the code model’s needs. In file localization, the same logic appears in repository-level code search methods that combine lexical retrieval over commit histories with neural reranking over code and messages. The broad lesson is that “retrieve more context” is no longer viewed as sufficient; the research trend is toward **query construction, selective retrieval, multi-stage reranking, and usefulness-aware filtering**. citeturn18view3turn21view1turn38view0turn21view6

A second line uses **graphs, dependencies, and static analysis** to make retrieval repository-aware. DraCo constructs a repo-specific context graph using extended dataflow analysis and retrieves context from that graph. GraphCoder builds a code context graph with control-flow and dependence edges and uses coarse-to-fine graph retrieval. STALL+ studies static-analysis integration directly and finds that file-level dependencies added in the prompting phase work better than some later-stage interventions. CodexGraph turns repositories into graph databases that agents can query structurally, aiming for code-structure-aware retrieval and navigation rather than pure text similarity. These methods all share the same premise: relevant context in a repository is often defined less by lexical similarity than by **symbol binding, dataflow, call structure, or dependency structure**. citeturn21view2turn21view4turn22view0turn21view5

A third line is **LSP-based or hybrid navigation**. The clearest published description comes from MarsCode Agent, which combines a repository knowledge graph, embeddings, keyword queries, reranking, and the Language Server Protocol for IDE-style “go to definition” and reference resolution. The paper is explicit that LSP is especially valuable where project-internal graph search is insufficient, such as overloaded symbols, ambiguous names, or references outside the target project. This makes LSP-style tools particularly relevant to your project’s concern with “does this agent configuration understand the concrete repository well enough to be trusted here?” because LSP gives a more realistic proxy for the sort of navigation a human developer would perform in an IDE. citeturn22view1

For **file localization and modification localization**, the literature traditionally splits between bug localization and fault localization. Bug localization typically starts from issue reports or bug reports and ranks relevant files; Bench4BL is the canonical large benchmark here, with 10,017 bug reports from 51 projects and standard MAP/MRR evaluation. Fault localization, by contrast, usually starts from failing and passing tests and produces suspicious statements, methods, or files, with long-running benchmarks such as Defects4J, Bears, and ManyBugs underpinning this line. Beyond debugging, the change-impact-analysis literature is older and broader: the classic surveys distinguish dependence-based CIA from traceability-based CIA and emphasize estimating the “estimated impact set” of a proposed change. The direct relation to your project is that issue-to-file retrieval, bug localization, and change impact analysis all provide candidate techniques for automatically generating *where to look* and *what should likely change* labels before patch generation is even evaluated. citeturn27view0turn30view2turn28search0turn28search1turn28search10turn29view1turn29view2turn21view6

## Common metrics for repository understanding

The literature uses several recurring metric families for “repository understanding,” but they are attached to different stages of the problem. For **retrieval and navigation**, the common metrics are Recall@k, Precision@k, Hit@k, MAP, MRR, and variants of ranking accuracy. ContextBench adds process-oriented measures such as context recall, context precision, and retrieval efficiency over agent trajectories. Traditional IR-style bug localization benchmarks such as Bench4BL rely heavily on MAP and MRR, while newer repository-level code search papers report MAP, MRR, and Precision@1 for issue-to-file retrieval. In practice, these metrics answer a narrow but important question: did the system surface the right files or code regions early enough to be useful? citeturn16view0turn30view2turn30view1turn21view7

For **completion and local code generation**, the standard metrics remain exact match, identifier-level match or F1, and executable success where available. CrossCodeEval and RepoBench evaluate retrieval-plus-completion settings where the central question is whether cross-file context improves code completion. RepoCoder reports improvements over in-file baselines on RepoEval, and DraCo and GraphCoder report exact match and identifier-focused gains when repository structure is used more precisely. ExecRepoBench pushes this family toward executable evaluation, which is closer to practical utility than text matching alone. These metrics are useful for “can the system exploit repository information to produce the correct code fragment,” but they are still weaker than end-to-end edit validation. citeturn17view5turn17view3turn18view0turn34view0turn21view4turn17view6

For **modification and maintenance tasks**, execution-based metrics dominate: pass@1 or pass@k on hidden tests, compile/build success, and repository-specific correctness checks. FEA-Bench, SWE-bench, SWE-PolyBench, and REPOFIXEVAL all evaluate some variant of executable correctness after repository edits. Newer task families add more structure-specific metrics because exact patch match is too brittle. DependEval uses fine-grained metrics for dependency reasoning and cross-file consistency. SWE-Refactor validates outputs through compilation, full tests, and automated refactoring detection, which is important because behavior preservation matters as much as textual similarity. RepoMod-Bench uses implementation-agnostic interface tests, and TimeMachine-bench emphasizes whether edits truly adapt the repository to dependency changes rather than merely gaming the visible tests. citeturn12view3turn13view5turn14view1turn19view5turn15view2turn26view3turn13view6turn13view7

For **fault localization**, the standard metrics are Top-k accuracy, EXAM score, and wasted effort. A recent large-scale SBFL evaluation makes the definitions explicit: EXAM measures the percentage of executable statements inspected before reaching the first faulty statement, Top-k measures whether the fault appears among the first k suspicious statements, and wasted effort counts how many non-faulty statements must be inspected first. These are highly relevant to repository-specific evaluation because a repository-level agent often fails *before* patch generation: it may never narrow the search to the right region cheaply enough. citeturn32view0

The most important recent metric lesson is that **passing tests is not enough**. A 2025 empirical study on SWE-bench Verified found that 7.8% of patches counted as correct still failed the developer-written test suite, and 29.6% of plausible patches behaved differently from the ground-truth patches under differential testing. That argues for behavioral-comparison metrics, not just visible-test pass rates, when the goal is to decide whether an agent configuration is trustworthy in a specific repository. citeturn35view0

## Gaps between benchmarks and real development

The first major gap is that many “repository-level” benchmarks still stop at **completion** rather than evaluating full software work. RepoBench, RepoEval, CrossCodeEval, and ExecRepoBench stress context use for local generation, but they usually do not require the agent to discover the right files, apply edits safely, manage repeated verification, or preserve whole-repository consistency after multi-file changes. By contrast, FEA-Bench and the SWE-bench family show that repository-level development often means implementing new functions or classes, modifying existing files, and surviving execution-based evaluation. The jump from completion to executable modification is exactly where many systems degrade. citeturn17view5turn18view0turn17view3turn17view6turn14view2turn12view3

The second gap is **benchmark freshness and representativeness**. SetUpAgent argues that manually curated suites like the original SWE-bench are bottlenecked by historically accurate environment setup, limiting them to a small number of popular repositories. Their automatically generated SWEE-Bench and SWA-Bench show substantial distributional differences, including more complex fixes and up to 40% lower agent success rates than on SWE-bench. SWE-bench-Live reaches a similar conclusion by moving to recent issues across 93 repositories, and SWE-QA-Pro explicitly targets long-tail repositories to reduce the chance that models can answer from memorized repository knowledge. This is directly relevant to your project because a repository-specialized benchmark generation method must be robust to **unpopular, idiosyncratic, fresh repositories**, not just “benchmark-famous” ones. citeturn20view0turn20view2turn20view3turn35view1turn25view0

The third gap is **validation fidelity**. Tests are an imperfect operationalization of developer intent. The SWE-bench patch-validation study shows that visible-test success can materially overstate true correctness. TimeMachine-bench likewise reports spurious solutions that exploit weak test coverage and unnecessary edits caused by poor tool use. For a project concerned with trust and permissions inside a specific repository, this is a crucial warning: end-task success on visible tests is an important signal, but it is an insufficient basis for claiming the agent “understands the repository” in the stronger engineering sense you care about. citeturn35view0turn13view7

The fourth gap is that most benchmarks do not explicitly model **repository conventions and workflow artifacts** beyond code and tests. Some come close: SWE-Refactor includes project structure, callers/callees, build configuration, and test coverage; FEA-Bench is mined from real pull requests for feature implementation; TimeMachine-bench and Bears use CI/build state; SetUpAgent mines issue resolution under historically accurate environments. But review comments, engineering norms, module ownership conventions, and task-specific permission boundaries are rarely first-class evaluation targets. Current benchmarks mostly operationalize correctness through retrieval labels, QA labels, compilation, tests, or AST/refactoring checks, not through repository-specific governance or review expectations. citeturn26view3turn14view2turn13view7turn28search17turn20view2

The fifth gap is **process visibility**. Benchmarks like SWE-bench tell you whether the final patch passed. They say much less about whether the system found the right files quickly, explored too broadly, relied on lucky execution feedback, or pulled in irrelevant context. ContextBench is notable precisely because it tries to fill this gap and finds that agent scaffolding yields only marginal improvement in context retrieval, that models often favor recall over precision, and that there is a substantial gap between the context they explore and the context they actually use. For evaluating an agent configuration rather than only a final patch, that is one of the strongest signals in the literature. citeturn16view0turn16view3

## Implications for repository-specific task generation

The literature suggests several concrete lessons for your project without prescribing a system design. First, **historical repository artifacts are already a proven source of executable tasks**. Issues and PRs support bug-fix and feature-implementation tasks in SWE-bench, SWE-PolyBench, and FEA-Bench. Dependency changes and CI failures support migration tasks in Bears, TimeMachine-bench, and SWE-bench-Live. Refactoring commits support behavior-preserving edits in SWE-Refactor. This means the raw materials you described—commit history, issues, PRs, tests, CI, and review traces—are aligned with how strong recent benchmarks are actually being built. citeturn12view3turn13view5turn14view2turn28search17turn13view7turn35view1turn26view3

Second, the best existing work increasingly separates **understanding**, **localization**, **editing**, and **verification** rather than collapsing them into one number. ContextBench contributes gold context and process metrics. Bench4BL gives file-level bug-localization oracles. DependEval makes dependency recognition and multi-file editing separate tasks. SWE-Refactor adds retrieval metadata such as callers and callees alongside executable validation. For a repository-specific benchmark generator, this strongly suggests that intermediate supervision or intermediate observables are where the current literature is richest and most reusable. citeturn16view0turn27view0turn15view4turn26view1

Third, the literature repeatedly warns against over-trusting **static shared benchmarks**. SetUpAgent, SWE-bench-Live, TimeMachine-bench, and SWE-QA-Pro all emphasize freshness, automation, or long-tail repository coverage as a defense against contamination, popularity bias, and overfitting. For your use case, that is especially relevant because a repository-specific benchmark generation method should derive value precisely from the *difference* between repositories. Current shared benchmarks can reveal broad trends, but they are not built to tell you whether one exact agent setup is trustworthy in *this* repo and *that* module. citeturn20view0turn20view3turn35view1turn13view7turn25view0

Fourth, there is a visible research gap. I did **not** find a widely adopted benchmark or method whose primary goal is: given one arbitrary real repository plus its history and workflow artifacts, automatically generate a replayable, graded, repository-specific benchmark for evaluating whether a full code-agent configuration deserves differentiated permissions inside that repository. There are strong partial precedents—automated environment setup, live benchmark curation, issue/PR mining, dependency-breakage mining, process-level context evaluation, and multi-file consistency benchmarks—but they stop short of the repository-specialized evaluation problem you described. That gap is precisely why the current literature is useful as a foundation rather than as a direct answer. citeturn13view8turn35view1turn13view7turn16view0turn15view4turn26view3
