## How existing work defines the evaluation target

A useful way to organize the literature is to separate four evaluation targets.

The first is **model-only or mostly-fixed-pipeline evaluation**. Repo-level benchmarks such as DevEval, RepoExec, and FEA-Bench evaluate models on repository-aware generation tasks while keeping the surrounding workflow relatively fixed. DevEval gives a full repository, requirements, and dependency annotations, then measures functional correctness and dependency recall. RepoExec emphasizes executability, dynamic test generation, and dependency use, but still evaluates under a controlled prompting/setup regime. FEA-Bench mines real pull requests from 83 repositories for feature implementation and verifies against paired tests, yet again primarily measures model performance inside a fixed benchmark pipeline rather than an open-ended agent workflow. These are important for repository realism, but they are not yet evaluations of a full operational Code Agent configuration. ŅłĆciteŅłéturn37view2Ņłéturn37view0Ņłéturn37view1Ņłü

The second is **agent-framework evaluation**. SWE-bench moved the field decisively in this direction by turning real GitHub issue resolution into an executable benchmark over a checked-out repository and hidden acceptance tests. The major conceptual shift is that the environment, action loop, and tool mediation become part of the test. AnthropicŌĆÖs SWE-bench write-up is unusually explicit here: SWE-bench evaluates the entire agent system rather than an isolated model, and performance can vary substantially with scaffolding even for the same model. The human-filtered SWE-bench Verified subset from ŅłĆentityŅłé["company","OpenAI","ai company"]Ņłü further tightened that framing by removing a portion of tasks whose solvability was confounded by missing context. ŅłĆciteŅłéturn10search9Ņłéturn34view0Ņłéturn4search12Ņłü

The third is **tool-augmented evaluation**, where the benchmark target is a model plus a particular tool interface and control loop. The SWE-agent paper is foundational here: its central claim is that interface design materially affects agent performance, and its implementation exposes configurable tools, action parsers, history processors, environments, and model budgets. Aider is similar in spirit though simpler in architecture: its own benchmark is end-to-end by design, measuring not only coding ability but also whether the model emits edits in a format the tool can reliably apply. These systems are closer to real deployment because they explicitly model the interaction between LLM outputs and operational tooling. ŅłĆciteŅłéturn20search12Ņłéturn20search18Ņłéturn28view0Ņłéturn28view1Ņłéturn29view0Ņłü

The fourth is **workflow- or system-level evaluation**. OpenHands Index fixes the OpenHands agent framework and then evaluates models by ability, cost, and runtime across five task families: issue resolution, greenfield apps, frontend work, software testing, and information gathering. CursorBench similarly treats the deployed coding workflow as the evaluation subject, using real internal sessions and measuring not just correctness but also quality, efficiency, and interaction behavior. ## Evidence that prompt, tools, retrieval, memory, and runtime change outcomes

The literature now contains several kinds of evidence that agent configuration is a first-class causal variable.

**Prompting and benchmark protocol matter.** A recent time-consistent repository-evaluation paper formalizes a matched A/B setup in which the same agent is evaluated with and without repository-derived knowledge while holding other factors fixed. In its baseline study on DragonFly and React, file-level F1 rises monotonically as prompts become more guided, reaching about 0.81 on both repositories for the strongest model, and the paper explicitly concludes that prompt construction is a first-order benchmark variable rather than packaging detail. This is directly relevant to your project because it shows that ŌĆ£same model, same repo, different prompt/granularityŌĆØ is already enough to require re-evaluation. ŅłĆciteŅłéturn23view1Ņłéturn23view2Ņłü

**Tool choice and tool mediation matter.** AiderŌĆÖs end-to-end code-editing benchmark found that plain text whole-file or diff-style edit formats outperformed function-calling edit formats, meaning that even when the underlying model is unchanged, the action encoding layer changes results. GitHub reports that reducing a default toolset from about 40 tools to a 13-tool core, together with embedding-guided routing and adaptive clustering, improved SWE-bench Verified and SWE-Lancer performance by 2ŌĆō5 percentage points and reduced latency. AnthropicŌĆÖs SWE-bench scaffold around Claude 3.5 Sonnet is intentionally minimalŌĆöprompt plus Bash tool plus Edit toolŌĆöbecause the company found that the benchmark outcome is a joint property of model and software scaffolding. ŅłĆciteŅłéturn29view0Ņłéturn32view1Ņłéturn34view0Ņłü

**Repository context helps only when it is high quality and relevant.** The AGENTS.md study is especially important because it tests a widespread industrial intuition directly. Across multiple agents and LLMs, repository-level context files tended to reduce task success rates while increasing inference cost by over 20%; they encouraged broader testing and exploration, and agents generally followed them, but often in ways that made tasks harder. By contrast, SWE-ContextBench shows that **correctly selected summarized context** can improve resolution accuracy and substantially reduce runtime and token cost, especially on harder tasks, while unfiltered or incorrectly selected context can have limited or negative value. A separate localization study shows that non-parametric repository memory over historical commits and linked issues can improve a strong localization framework. Together these results imply that retrieval and memory are not monotonic ŌĆ£more is betterŌĆØ knobs; they are quality-sensitive workflow components that need repository-specific evaluation. **Runtime budgets and infrastructure settings matter.** AnthropicŌĆÖs infrastructure-noise analysis is one of the clearest demonstrations: on Terminal-Bench 2.0, the gap between the most- and least-resourced setups was 6 percentage points under the same Claude model and same harness, with the runtime treated as an integral part of problem-solving rather than a passive container. ## How the field is starting to evaluate agent trajectories rather than only final patches

A major recent shift is from **outcome-only evaluation** to **process-level evaluation**.

RACE-bench is representative of this shift. Instead of treating repository-level feature addition as a black box, it attaches structured intermediate reasoning ground truth for issue understanding, file localization, implementation tasks, and step decomposition, and jointly scores patch correctness and reasoning quality. Its main empirical result is not just that resolved rate ranges widely across agents, but that reasoning quality degrades in a waterfall pattern: agents are relatively better at high-level understanding than at turning intent into concrete implementation steps. This maps directly onto the trajectory components you listed: planning quality, localization, decomposition, and execution. ŅłĆciteŅłéturn38view0Ņłéturn38view1Ņłü

SWE-eval takes a more generic trajectory perspective. It adds metrics for efficiency, logical consistency within and across turns, and tool utilization via an information-gain measure. Its authors argue that trajectory-aware metrics expose failure modes hidden by final resolution rate alone, including shallow exploration, missing backtracking, and loop entrapment. This is precisely the kind of evidence your project needs if the goal is to judge whether a workflow is trustworthy inside a repository rather than merely capable of sometimes producing a passing patch. ŅłĆciteŅłéturn22view9Ņłéturn22view10Ņłü

ContextBench unboxes another central part of the trajectory: context selection. It creates human-verified gold contexts for 1,136 issue-resolution tasks across 66 repositories and 8 languages, instruments agents to trace which code regions they inspect, and compares retrieved context against those gold contexts. The paper finds notable gaps between retrieved context and actually used context, as well as diverse redundancy/efficiency trade-offs among frontier models. That makes it one of the strongest current answers to ŌĆ£how do we evaluate file localization and context selection?ŌĆØ ŅłĆciteŅłéturn23view4Ņłéturn23view5Ņłü

Failure-analysis work complements these benchmarks. A manual study of 150 failed SWE-bench Verified cases shows distinct failure fingerprints across architectural paradigms and emphasizes flawed reasoning and cognitive deadlocks in interactive agents. The same paper also shows that multi-file coordination exposes larger differences than single-file tasks, with OpenHands degrading less sharply than Agentless or standard-scaffold Tools Claude. Related environment-focused work such as EnConda-bench pushes process-level evaluation into repository setup itself, assessing planning, perception-driven diagnosis, repair, and final action during environment configuration. ## What this means for repository-specific benchmark generation and repository-specific admission

The closest public precursor to your problem statement is **SetUpAgent**. Its explicit goal is to generate SWE-Bench-like benchmarks automatically from arbitrary repository lists, with historically accurate dependency setup, test execution, and result parsing. The most important result is not just automation itself, but the observation that newly generated benchmarks can differ substantially from SWE-bench in issue quality, fix complexity, and agent performance, with up to 40% lower success rates. That is a strong empirical argument that a fixed benchmark family does **not** adequately stand in for arbitrary repositories. ŅłĆciteŅłéturn25view0Ņłéturn25view1Ņłéturn25view3Ņłü

The second key precursor is the **time-consistent benchmark for repository-level software engineering evaluation**. It snapshots a repository at time T0, constructs repository-derived knowledge using only pre-T0 artifacts, and evaluates on pull requests merged later, while comparing the same agent with and without the repository-derived knowledge under matched conditions. This is methodologically very close to what a repository-specific evaluation program needs, because it explicitly controls temporal leakage and holds non-target variables fixed. It also shows that even under such careful design, prompt granularity remains a major source of variance. ŅłĆciteŅłéturn23view1Ņłéturn23view2Ņłü

Other benchmark-generation papers provide useful pieces. SWE-MERA auto-generates dynamic software-engineering tasks by identifying future tests that fail in an earlier repository snapshot and then re-running the repository under that state. TTG-GEN automatically creates repository-level comprehension tasks with low direct solution leakage using coverage-guided fuzzing. Long Code Arena and related benchmark suites mine source code, commit history, issues, and build data from GitHub repositories to create diverse long-context tasks. SWD-Bench mines high-quality pull requests and repository context to evaluate documentation, and reports that stronger documentation measurably improves an issue-solving agentŌĆÖs success rate. These works collectively show that repository artifacts can be transformed into scalable evaluation tasks; they do **not** yet add up to repository-specific certification of complete agent workflows. ŅłĆciteŅłéturn37view4Ņłéturn12search0Ņłéturn12search3Ņłéturn12search2Ņłü

So, can current evaluations support a real ŌĆ£given this repository, should this agent configuration be admitted?ŌĆØ decision? **Partially, but not yet robustly.** They now support several necessary ingredients: historically faithful task construction, repo-specific data splits, process-level diagnosis, context-quality measurement, and cost/runtime reporting. But most public benchmarks are still either static, cross-repository averages, or fixed-framework leaderboards. They are not designed to make high-stakes, repo-scoped trust decisions about a concrete bundle of model, prompt, tools, memory, rules, and permissions. There is essentially no established public work that turns repository-specific performance into certification or gating in the narrow sense you described. ## The main research gaps

- **Few papers truly treat the full agent configuration as the evaluation object.** Many papers say they study ŌĆ£agents,ŌĆØ but in practice they either compare models under one scaffold or compare a few complete systems without systematic factorization of prompt, tools, memory, permissions, and runtime. - **Systematic ablations remain sparse.** There is growing evidence that prompt format, tool selection, edit format, retrieval quality, memory quality, and resource budgets matter, but there are still relatively few controlled studies that vary these factors one at a time on the same repository and the same task family. The AGENTS.md study, SWE-ContextBench, AiderŌĆÖs edit-format benchmark, and the time-consistent benchmark are valuable precisely because they start doing this. - **Repository-specific heterogeneity is under-modeled.** SetUpAgent shows large distributional shifts when benchmarks expand beyond SWE-benchŌĆÖs repository set. ŅłĆciteŅłéturn25view0Ņłéturn36view0Ņłéturn36view4Ņłéturn10search1Ņłü

- **Trajectory benchmarks are still fragmented.** RACE-bench, SWE-eval, ContextBench, and EnConda-bench each illuminate different slices of agent behaviorŌĆöplanning, localization, context retrieval, tool efficiency, environment setupŌĆöbut there is no dominant unified framework that combines these into a standard, repository-scoped trust assessment. - **Temporal validity and leakage control are only recently being treated as core requirements.** The time-consistent benchmark is an important methodological advance, but this design principle is not yet standard across repository-aware evaluations.
