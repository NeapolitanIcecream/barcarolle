# Environment Replay and Reproducible Execution for Software Engineering Task Evaluation

## Why this literature matters for repository-specific benchmark generation

The closest prior work to your problem does **not** come from a single benchmark family. It comes from four partially overlapping lines of work: repository-level issue-resolution benchmarks such as SWE-bench and its descendants; automated benchmark-generation systems such as SetUpAgent and SWE-bench-Live/RepoLaunch; executable repository environments such as R2E and SUPER; and interactive agent harnesses such as Terminal-Bench, Harbor, OSWorld, OpenHands, and SWE-agent that emphasize sandboxing, verification, and trace capture. Taken together, these works show that the hard part is usually **not** task mining aloneÔÇöit is building a historically faithful execution environment, selecting trustworthy tests or verifiers, and deciding what to do when reality diverges from the reconstructed environment. ţłÇciteţłéturn31view0ţłéturn8view0ţłéturn33view0ţłéturn29view0ţłéturn39view0ţłéturn12view0ţłéturn18view1ţłéturn24view0ţłéturn22view0ţłü

A useful framing distinction is that most existing systems evaluate one of three things: **issue resolution against historical repository tests** (SWE-bench-style), **task execution in a live or synthetic interactive environment** (Terminal-Bench, OSWorld), or **generated executable harnesses at function/repository scope** (R2E, SUPER). Your targetÔÇö*automatically generating repository-specific, replayable, graded tasks from a repositoryÔÇÖs own history and artifacts*ÔÇösits between these categories. The literature is therefore highly relevant, but no single prior system already solves the full problem. ţłÇciteţłéturn31view0ţłéturn33view0ţłéturn29view0ţłéturn39view0ţłéturn12view0ţłéturn18view1ţłü

## Main technical difficulties in environment replay

The central technical obstacle is **time fidelity**. A repository commit does not fully determine a runnable environment because package registries, base images, hosted CI runners, and system packages all change over time. SetUpAgent addresses this by constraining Python dependency installation to versions released before the issue date; RepoLaunch does something similar with a time-filtering package proxy; snapshot archives such as the one maintained by ţłÇentityţłé["organization","Debian","linux distribution project"]ţłü expose historical packages by date and version; and reproducible-builds work formalizes `.buildinfo` as a record of build-environment inputs. These techniques all converge on the same lesson: **historical replay requires temporal metadata, not just package names**. ţłÇciteţłéturn30view1ţłéturn33view0ţłéturn7view12ţłéturn37search0ţłéturn37search1ţłü

Lockfiles help, but in uneven ways across ecosystems. The lockfile literature emphasizes that lockfiles record resolved direct and indirect dependencies, improve reproducibility, and often include integrity data, but support differs substantially across package managers. The recent Maven-Lockfile work is interesting precisely because it retrofits direct/transitive dependency capture and checksums into an ecosystem that historically lacked native lockfiles, making past rebuilds easier. This is highly relevant for repository-specific benchmark generation because a generic pipeline cannot assume every repository has a high-quality lockfile, or even that ÔÇťlockfileÔÇŁ means the same thing across languages. ţłÇciteţłéturn38view0ţłéturn7view15ţłü

Containers are helpful but not sufficient. The reproducibility literature has become more explicit on this point: Dockerized execution is a strong packaging mechanism, but rebuilding a Docker environment later can still drift if package repositories, transitively fetched artifacts, or mutable base tags have changed. The Nix study shows that functional package management can deliver extremely high rebuildability over time, while the recent ÔÇťDocker Does Not Guarantee ReproducibilityÔÇŁ paper argues that containers alone do not solve time-dependent dependency resolution. For benchmark generation, this means that ÔÇťwe have a DockerfileÔÇŁ is much weaker evidence than ÔÇťwe have an image digest, archived package sources, and a record of all resolved artifacts.ÔÇŁ ţłÇciteţłéturn7view14ţłéturn7view16ţłéturn28view0ţłü

CI configuration is simultaneously one of the best clues and one of the weakest sources of ground truth. SetUpAgent and RepoLaunch both lean heavily on CI/CD files because they often reveal the right Python version, services, test entrypoints, and environment variables. But literal CI replay is difficult because hosted CI labels are mutable, and artifacts/logs may expire. For example, the `ubuntu-latest` label in GitHub-hosted runners rolled to Ubuntu 24.04 in 2025, and GitHubÔÇÖs default retention for workflow artifacts and logs is 90 days. In practice, most benchmark systems therefore perform **semantic CI replay**ÔÇöthey infer the intended environment from CI filesÔÇörather than replaying the original hosted run bit-for-bit. ţłÇciteţłéturn30view2ţłéturn33view0ţłéturn7view10ţłéturn6search16ţłéturn6search14ţłü

System-level dependencies and service topology are another recurring failure point. GitHub Actions supports job containers, sibling container actions, mounted volumes, and service containers, which means the actual execution context may include databases, caches, browsers, or additional networked services that are not represented in a language lockfile. SetUpAgentÔÇÖs own failure analysis found many cases where the only documented way to test the application required Docker containers, and those cases often failed under its setup regime. This is important for repository-specific benchmark generation: app-style repositories tend to require **multi-process or multi-service replay**, not just package installation. ţłÇciteţłéturn7view10ţłéturn30view7ţłü

Finally, there is a **granularity mismatch** across the literature. R2E demonstrates that one can automatically convert many repositories into executable environments using generated equivalence tests, but its target is function/method evaluation rather than reconstruction of a repositoryÔÇÖs historical issue-fix state. SUPER shows that setup-and-execute tasks are meaningful benchmark material, yet its evaluation is closer to ÔÇťcan the agent make the research repo runÔÇŁ than ÔÇťcan we faithfully replay a historical repository state and validate a code change against historical expectations.ÔÇŁ For your project, both are relevant inspirations, but neither is a drop-in template. ţłÇciteţłéturn29view0ţłéturn29view3ţłéturn39view0ţłü

## How existing benchmarks recover, run, and select tests

SWE-bench-style systems operationalize correctness through **state transitions in repository tests**. In the original setup, the benchmark is built from issue/PR pairs where the PR modifies tests; after reconstructing the pre-fix environment, the benchmark checks whether tests that should move from failing to passing actually do so. SWE-bench-Live makes this formulation explicit with both **FAIL_TO_PASS** and **PASS_TO_PASS** fields, and SetUpAgent uses full-suite execution before and after the gold patch to derive those expected behaviors and then filters out any PR that lacks at least one FÔćĺP test. This is a strong pattern for repository-derived benchmark generation because it yields tasks grounded in real developer-written regression tests rather than synthetic correctness criteria. ţłÇciteţłéturn31view0ţłéturn33view0ţłéturn30view5ţłü

A notable contrast is that interactive environment benchmarks often avoid per-test selection in favor of **custom verifiers**. Terminal-Bench tasks package a Docker environment plus a verifier script, and Harbor copies `/tests` into the environment at runtime and expects the verifier to write a numeric reward file. This design is useful because it supports much richer success criteria than ÔÇťdid `pytest` pass,ÔÇŁ including multi-step workflows, server setup, file production, or mixed functional/behavioral checks. It also gives benchmark builders the option of keeping verifier implementation outside the agentÔÇÖs working tree, which is effectively a hidden-test mechanism. ţłÇciteţłéturn12view0ţłéturn28view0ţłü

OSWorld pushes this further by using **execution-based evaluators over prepared machine states**. Each task example comes with an initial state setup and a custom evaluator; success is determined by whether the post-run machine state satisfies the scripted criteria. For repo-specific software tasks, that suggests a broader lesson: once the task extends beyond unit tests into services, browsers, or full-stack workflows, repository-derived tests may need to be complemented by environment-level evaluators. ţłÇciteţłéturn18view1ţłéturn18view0ţłü

R2E uses a different strategy again: it generates **equivalence-test harnesses** from repository code and filters them by validity and coverage. This is not suitable for direct issue-resolution replay, but it is a strong reference for automated test generation when a repository has insufficient direct historical test signal for some subtask. The key insight is that generated tests are only useful if they are themselves executable, quality-filtered, and grounded in repository semantics rather than surface-level I/O guessing. ţłÇciteţłéturn29view0ţłéturn29view1ţłéturn40view2ţłü

Flakiness is handled more by **benchmark curation policy** than by sophisticated statistical testing. SWE-bench-Live says it performs multiple rounds of test execution to ensure the task consistently exhibits the expected behavior. Terminal-Bench 2.0 was explicitly motivated by cases where tasks from the earlier release drifted over timeÔÇöfor example, because an external website changed behaviorÔÇöso the maintainers re-verified tasks with substantial manual and LM-assisted effort. OSWorld-Verified likewise exists because community reports surfaced problematic tasks and signals. In other words, the dominant pattern today is: **detect instability through repeated execution and community use, then repair/filter/retire tasks**. ţłÇciteţłéturn33view0ţłéturn12view4ţłéturn18view1ţłü

## How execution is recorded and how irreproducible samples are treated

The trace formats in current systems are rich enough to support auditing, but they are fragmented. SWE-agentÔÇÖs `.traj` format stores the step-by-step response, parsed thought, action, observation, environment state, and the exact query shown to the model; it also emits configuration files, predictions, logs, and exit-status summaries. The public SWE-bench experiments repository goes further by releasing predictions, execution logs, trajectories, and results, and it requires reasoning traces for leaderboard submissions. For any repository-specific benchmark intended to establish whether a *particular agent configuration* is trustworthy, this kind of artifact capture is probably closer to the minimum acceptable standard than a luxury. ţłÇciteţłéturn22view0ţłéturn7view9ţłü

Terminal-Bench and Harbor record execution in a more systems-oriented way. Trial output can include session data, terminal panes before/after execution, `commands.txt`, `results.json`, agent logs, and verifier logs under `/logs/verifier/`; Harbor also standardizes agent trajectories using ATIF. This is especially valuable for replay because it separates **what the agent did** from **how the verifier judged it**, which makes post-hoc debugging much easier. ţłÇciteţłéturn16view0ţłéturn28view0ţłéturn28view1ţłü

OpenHands emphasizes persistent conversation state. Its persistence layer stores message history, agent configuration, execution state, tool outputs, working-directory context, secrets metadata, and event logs under a per-conversation directory structure, and it exposes a trajectory API. That makes OpenHands relevant not because it reconstructs historical repository environments, but because it treats replayable agent state as a first-class object. ţłÇciteţłéturn24view2ţłéturn24view3ţłü

OSWorld records a different artifact mix: screenshots, action traces, and video output in the results directory, with a separate manual-examination path for replaying and inspecting problematic tasks. This is a reminder that ÔÇťreplay artifactsÔÇŁ should match the environment modality: shell tasks need commands and stdout; GUI tasks need state snapshots and visual recordings. ţłÇciteţłéturn19view1ţłéturn19view3ţłü

On irreproducible samples, the common strategies are **filtering, manual verification, environment patching, and benchmark maintenance** rather than trying to salvage every case. SetUpAgent filters out repositories for which environment setup fails and PRs that do not exhibit the required fail/fix behavior; in one reported pipeline, only a minority of candidate repositories survived all filters through successful setup. SWE-bench Verified addresses quality through human filtering of 500 instances. Terminal-Bench 2.0 and OSWorld-Verified represent explicit maintenance passes after community feedback. SWE-bench-Live goes further and makes dynamic refresh part of the benchmark concept itself. ţłÇciteţłéturn30view5ţłéturn31view1ţłéturn12view4ţłéturn18view1ţłéturn33view0ţłü

There is also a smaller but important line of work on **reproducible dependency-failure corpora**. BUMP stores reproducible breaking dependency updates as Dockerized artifacts and guarantees consistent build failures over time, while ComBench requires that an original CI compilation failure be replayable locally in a faithful environment and then validated end-to-end through failÔÇôfixÔÇôpass execution. These works are not issue-resolution benchmarks in the SWE-bench sense, but they are among the clearest examples of treating reproducibility itself as a benchmark invariant rather than an implementation detail. ţłÇciteţłéturn36view0ţłéturn7view17ţłü

## Implications for generating benchmarks from a given repository

The literature strongly suggests that a good repository-specific generation pipeline should treat **environment recovery**, **task extraction**, and **task validation** as three separate stages. SetUpAgent and RepoLaunch both adopt this separation, and ComBenchÔÇÖs failÔÇôfixÔÇôpass replay logic reinforces it. Without this separation, it becomes too easy to confuse ÔÇťthe agent failed the taskÔÇŁ with ÔÇťthe benchmark could not faithfully reconstruct the task.ÔÇŁ ţłÇciteţłéturn30view2ţłéturn33view0ţłéturn7view17ţłü

It also suggests that environment construction should be **evidence-fusing**, not single-source. The strongest recoverable signals come from combining manifests/lockfiles, Dockerfiles, CI/CD config, README/CONTRIBUTING text, package-manager metadata, and historical archives. The reproducible-builds ecosystem adds a further lesson: recording full build metadata (`.buildinfo`, archived package sources, checksums, image digests) is far more future-proof than relying on best-effort re-resolution. This is particularly relevant if the benchmark will need to be re-run months or years later to decide whether an agent configuration remains trustworthy on a repository. ţłÇciteţłéturn30view2ţłéturn7view12ţłéturn37search0ţłéturn37search1ţłéturn38view0ţłéturn7view15ţłü

Another strong implication is that **benchmark membership should be gated by reproducibility labels**. Existing work repeatedly distinguishes at least four statuses: setup succeeds or fails; verification runs or fails; expected fail/pass transitions exist or do not; and the task is stable or flaky across repeated runs. SetUpAgent, SWE-bench-Live, Terminal-Bench 2.0, and OSWorld-Verified all in different ways operationalize this idea. For your project, that matters because repository-specific authority decisions depend more on benchmark trustworthiness than on benchmark size. ţłÇciteţłéturn30view5ţłéturn33view0ţłéturn12view4ţłéturn18view1ţłü

The work on trajectories further implies that evaluating a *full agent configuration* requires storing enough artifacts to support **audit replay**, not just final pass/fail scores. SWE-agent, Harbor, and OpenHands all show practical patterns here: record shell commands, tool invocations, file-edit context, event histories, model inputs at each step, verifier outputs, and environment state. This aligns unusually well with your ÔÇťagent licenseÔÇŁ framing, because the question is not only ÔÇťdid it pass?ÔÇŁ but ÔÇťwhat exactly did this agent configuration do in this repository to pass?ÔÇŁ ţłÇciteţłéturn22view0ţłéturn28view1ţłéturn24view3ţłü

A final implication is that **continuous benchmark maintenance is not optional**. Mutable external dependencies, CI image drift, expiring artifacts, changing web services, and community-discovered edge cases all show up repeatedly across the literature. SWE-bench-Live makes freshness a central feature, Terminal-Bench 2.0 was driven by drift-induced task problems, and OSWorld-Verified formalizes benchmark repair. So the most transferable lesson is not a particular toolchain but a process principle: repository-specific executable benchmarks should be maintained as living assets, with explicit retirement/repair rules for broken tasks. ţłÇciteţłéturn33view0ţłéturn12view4ţłéturn18view1ţłéturn6search16ţłéturn6search14ţłü
