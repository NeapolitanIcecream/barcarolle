# Related Work on Repository-Specific Benchmark Generation for Code Agents

The literature closest to your project falls into four partially overlapping strands: executable software-engineering benchmarks mined from historical repository activity, repository-level code understanding benchmarks, interactive agent environments, and automatic benchmark/environment construction pipelines. The papers most directly adjacent to your actual problem are not the original fixed suites, but the generation-oriented line—especially R2E, SetUpAgent, FeatureBench, SWE-bench-Live, SWE-rebench, SWE-MERA, SWE-Mirror, and SWE-Universe—because they start from a repository or pull-request stream and try to produce executable evaluation tasks automatically. ## Major technical routes

The first and most established route is **historical issue/PR replay**: mine an issue, recover the repository state before the fix, hide the tests or test deltas, and score a generated patch by executing the repository’s verification suite. SWE-bench made this pattern standard; its descendants mainly vary task diversity, oracle quality, modality, language coverage, and contamination resistance. SWE-bench Verified adds human filtering, SWE-bench Multimodal brings visual JavaScript tasks, SWE-bench Pro raises task horizon and repository diversity, Multi-SWE-bench expands across languages, SWE-Lancer brings end-to-end and managerial tasks, and SWE-bench-Live refreshes the benchmark continuously. citeturn19view0turn17view1turn17view2turn17view3turn19view3turn17view5turn19view5

A second route is **repository-level comprehension without full software replay**. RepoBench, CrossCodeEval, RepoQA, DependEval, and DevEval each isolate one bottleneck: cross-file retrieval, cross-file completion, long-context understanding, dependency reasoning, or repository-aligned code generation. This line is useful because it produces controllable diagnostics; it is limited because it usually stops short of replayable history, hidden-oracle verification, and interactive development trajectories. ContextBench is the most important bridge work here because it stays in issue resolution but adds human-annotated gold context and trajectory-level retrieval metrics. citeturn16view0turn15view0turn15view3turn30view3turn16view7turn16view2

A third route is **general agent environment design**. AgentBench, WebArena, OSWorld, Terminal-Bench, and GAIA are not repository benchmarks, but they matter methodologically because they show how to make interactive tasks reproducible, long-horizon, and tool-using. WebArena and OSWorld are especially relevant for environment replay and execution-based scoring; Terminal-Bench is relevant for neutral harness design around a constrained interface; GAIA is relevant for benchmark freshness and hidden-answer leaderboard structure. This whole line evaluates workflows more directly than most software benchmarks, but its task semantics are mostly not repository-specific. citeturn25view7turn14view0turn25view6turn25view0turn25view1turn30view0

The fourth route is **automatic environment and task construction**. R2E turns any repository into executable function-level environments using program analysis plus LLM-generated equivalence tests. SetUpAgent automates historically accurate dependency setup and result parsing for issue-based benchmarks. FeatureBench automatically derives feature-oriented tasks by tracing from tests through dependency graphs across multiple commits and PRs. SWE-rebench and SWE-bench-Live make the pipeline continuous and decontaminated. SWE-MERA, SWE-Mirror, SWE-Bench++, and SWE-Universe push still further toward automated, rolling, or massive-scale generation. This is the route most aligned with your project, but most of these works optimize for benchmark scale, training data, or public rolling evaluation rather than for building a one-repository accreditation artifact. citeturn13view5turn24view2turn14view6turn24view5turn19view5turn19view7turn24view6turn22search0turn24view4

The fifth route is **benchmark hardening and richer oracles**. The strongest lesson from the last two years is that “tests pass” is not enough. SWE-Bench+ found direct solution leakage and weak tests in many SWE-bench successes; the 2026 entity["company","OpenAI","ai research company"] audit of SWE-bench Verified found many residual flawed tests in the unsolved portion; AntiLeakBench argues that merely collecting recent data does not guarantee decontamination; SWE-Lancer replaces unit tests with professional end-to-end tests where possible; ContextBench and RACE-bench add intermediate process signals; ST-WebAgentBench and DepDec-Bench show how policy adherence and hidden risk can be measured beyond final success. citeturn21view1turn21view0turn21view4turn29view4turn16view2turn27view0turn26view0turn29view3

## Research lessons

1. **Realistic tasks come most naturally from historical development traces, not from synthetic prompts.** The most credible software benchmarks all anchor tasks in issues, PRs, repositories, or real engineering work, because that preserves ambiguity, long-range context, and real change structure. citeturn19view0turn17view3turn20view0turn17view5

2. **Environment synthesis is the main bottleneck for scaling repository-level evaluation.** SetUpAgent, SWE-bench-Live, SWE-rebench, and SWE-Universe all treat historically accurate setup and executable verification as the core engineering challenge rather than a peripheral detail. citeturn24view2turn19view5turn24view5turn24view4

3. **Bug fixing is only one slice of repository work.** FeatureBench, FEA-Bench, RACE-bench, SWE-bench Multimodal, and SWE-Lancer all show that feature development, visual debugging, end-to-end product tasks, and even managerial decision-making expose very different failure modes from classic issue-to-patch repair. citeturn14view6turn28search0turn27view0turn17view2turn17view5

4. **Intermediate process signals are indispensable.** ContextBench shows that context retrieval can be measured directly; RACE-bench shows that reasoning stages can be scored separately; both suggest that benchmark generation should produce more than a final hidden test suite. citeturn16view2turn27view0

5. **Dynamic freshness is not optional anymore.** SWE-bench-Live, SWE-rebench, SWE-MERA, AntiLeakBench, and leakage studies all converge on the same conclusion: once a benchmark becomes public and important, contamination pressure grows quickly, so freshness must be built into the pipeline. citeturn19view5turn24view5turn19view7turn21view4turn21view3

6. **Hidden-oracle quality matters as much as task realism.** SWE-Bench+, the 2026 OpenAI audit, and SWE-Lancer all show that weak, overly narrow, or gameable tests distort measured capability badly. citeturn21view1turn21view0turn29view4

7. **Language and ecosystem diversity materially change conclusions.** Multi-SWE-bench and SWE-bench Pro both show that Python-centric public suites can overstate how well methods generalize across languages, repository styles, and business domains. citeturn19view3turn17view3

8. **Benchmarking whole systems is necessary, but current practice treats the system as a black box.** SWE-bench Verified leaderboards compare everything from simple loops to RAG and multi-review systems; Terminal-Bench and general agent benchmarks provide fairer harnesses; yet none of them makes prompt rules, tool permissions, retrieval policies, or memory strategy first-class benchmark variables. citeturn25view4turn25view3turn25view1turn25view7

9. **Different generation methods cover different granularities of repository work.** R2E is strong for turning arbitrary codebases into executable function environments, while FeatureBench is strong for multi-commit feature tasks and SetUpAgent is strong for issue-based historical replay. A repository-specific benchmark generator will likely need all three granularities available, even if it does not combine them in one score. citeturn13view5turn14view6turn24view2

10. **Trustworthy collaboration needs metrics beyond functional correctness.** ST-WebAgentBench formalizes policy-aware completion, DepDec-Bench measures dependency-risk decisions invisible to tests, and SWE-Lancer relates performance to economic impact; together they suggest that admission-style evaluation should include safety, restraint, and consequence sensitivity, not only patch success. citeturn26view0turn29view3turn17view5

## Research gaps

The clearest gap is **given-repository benchmark generation**. Several works can generate environments or harvest tasks at scale, but they do so to build shared datasets, public benchmarks, or training corpora. R2E works at function granularity; SetUpAgent generates broad issue-based corpora; FeatureBench derives feature tasks across repositories; SWE-bench-Live, SWE-rebench, and SWE-MERA build rolling public evaluations; SWE-Mirror and SWE-Universe push generation volume even further. None of these is framed as: “take one repository plus its full engineering history, and automatically produce a graded benchmark specialized to that repository’s modules, workflows, and norms.” That conclusion is an inference from how these papers define their outputs. citeturn13view5turn24view2turn14view6turn19view5turn24view5turn19view7turn24view6turn24view4

A second gap is **agent-configuration-level evaluation**. Existing benchmarks usually compare either whole systems against one another or language models under a normalized minimal scaffold. SWE-bench Verified explicitly separates a full-system leaderboard from a bash-only mini-SWE-agent track; Terminal-Bench provides a reusable harness; AgentBench, WebArena, and OSWorld evaluate interactive agents. But none of them elevates the full agent configuration—model choice, scaffold, tools, permissions, prompts, rule set, memory policy, retrieval strategy, and execution sandbox—to the status of the object being benchmarked. The benchmarked unit is still usually “a submission” or “an agent,” not a compositional configuration space. citeturn25view3turn25view4turn25view1turn25view7turn14view0turn25view6

A third gap is **repository-specific trusted collaboration**. Most software benchmarks ask whether an agent can solve a task, not whether it can be trusted in a particular repository under realistic operational constraints. ContextBench and RACE-bench partially move in this direction by looking inside trajectories, but they still stop short of task-type-, module-, or authority-sensitive trust. ST-WebAgentBench and DepDec-Bench are the nearest conceptual neighbors because they score policy compliance and hidden operational risk, yet they do so outside repository-specific software-governance settings. citeturn16view2turn27view0turn26view0turn29view3

A fourth gap is **mapping evaluation to permission governance**. Existing work reports resolved rate, Pass@k, retrieval quality, success rate, monetary value, CuP, or risk ratios. What is missing is a validated mapping from repository-specific benchmark outcomes to permission decisions such as “may edit tests but not production code,” “may work in module A but not B,” or “may propose PRs but not merge without review.” SWE-Lancer gets closest by tying outcomes to economic value; ST-WebAgentBench gets closest by tying outcomes to policy compliance; neither produces a repository-scoped authority model. citeturn17view5turn26view0turn29view3

A fifth gap is **use of richer repository history beyond issue text and tests**. Current generation pipelines mainly consume issues, PR metadata, repository snapshots, tests, CI/CD config, READMEs, docs, and dependency files. SetUpAgent explicitly extracts commands from repo files, CI/CD configs, referenced webpages, and similar versions; Multi-SWE-bench builds from PR metadata, Dockerized reproduction, filtering, and manual verification; FeatureBench traces from tests and dependency graphs; SWE-bench itself is issue/PR/test centered. Review discussions, coding conventions, repository-specific engineering rules, and authorization policies are still not first-class inputs in benchmark generation pipelines. That absence matters directly for your problem because those artifacts are precisely where “trustworthy collaboration” is often encoded. citeturn4search5turn20view0turn14view6turn19view0
