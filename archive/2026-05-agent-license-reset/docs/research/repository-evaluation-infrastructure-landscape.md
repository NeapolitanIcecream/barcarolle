## Harnesses, logging, and safety controls

SWE-agent remains the clearest reference for a shell-first software-engineering agent harness. Its contribution is not just prompting, but a custom agent-computer interface around repository search/navigation, file viewing/editing, and command execution. The environment defaults to Docker, and the main artifact is the trajectory, which the project treats as the primary output and exposes through both CLI and web inspectors. ## Prototype versus production patterns

Research prototypes overwhelmingly optimize for **fidelity to repository-native commands** and **low friction**. The common recipe is plain Docker, local git history, shell execution, repo-native tests, and trajectory/log files stored on disk. SWE-agent, Aider, GitBug-Actions, and many environment-setup benchmarks all fit this pattern. It is the fastest way to achieve execution-grounded evaluation and the easiest way to publish reproducible artifacts. ţłÇciteţłéturn30search0ţłéturn31search0ţłéturn25search2ţłéturn38search1ţłü

## Research gaps and key references

The largest research gap is **multi-source repository task synthesis**. Public pipelines typically choose one principal evidence stream: issues for SWE-bench-style tasks, PR merges for SWE-Next/SWE-Bench++, CI histories for GitBug-Actions and ComBench, or synthetic test breaking for SWE-smith. I did not find a public system that systematically combines commit history, issue/PR text, review threads, CI definitions, test suites, and repository conventions into one unified on-demand repository-specific benchmark generator. That is a genuine gap, not just a missing feature. ţłÇciteţłéturn35search0ţłéturn25search0ţłéturn36search0ţłéturn25search2ţłéturn25search8ţłéturn26search2ţłü

The next gap is **historically accurate environment replay at acceptable cost**. Recent work now treats this as the bottleneck: Repo2Run targets reliable Docker environment configuration; EnvBench and Multi-Docker-Eval show low-to-moderate success even for strong agents; DockSmith exists because environment construction transfers to broader agent performance; GitBug-Actions and ActionsRemaker focus on faithful CI reconstruction; SWE-Next introduces environment reuse because na├»vely building/storing repository-specific environments dominates system cost. In other words, the field has already discovered that ÔÇťgetting runnableÔÇŁ is often harder than ÔÇťediting the code.ÔÇŁ ţłÇciteţłéturn39search3ţłéturn38search0ţłéturn37search0ţłéturn39search2ţłéturn25search2ţłéturn10search0ţłéturn25search0ţłü

There is also a clear gap in **artifact standardization**.
